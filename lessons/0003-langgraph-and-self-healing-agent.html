Lesson 3 · Agentic Workflows & State Machines

LangGraph Orchestration & Self-Healing Loops

LangGraph turns a one-shot code generator into a self-correcting loop. This lesson explains the state machine, the Predict → Act → Verify cycle, and the multi-language runtime challenges that had to be solved.

In standard software development, code generation is a one-off attempt — the LLM outputs code and if it fails to run, the user fixes it manually. In agentic engineering we build a state-transition engine that automates verification and healing. LangGraph is the framework that makes that possible.

Understanding LangGraph State Machines

LangGraph lets us define agent workflows as a cyclic directed graph. The graph shares a persistent state dictionary (AgentState). Each node is a function that reads the state, does work, and returns updated fields.

The Shared State — AgentState

Every piece of execution context lives in one typed dictionary, passed between all nodes without any implicit global state.

Key fields:
  • scraped_text — the raw documentation context
  • use_case — the developer's stated goal
  • language — target language (Python, Go, Java…)
  • code & tests — the current generated iteration
  • error_logs — compiler stdout/stderr from the last run
  • retry_count — controls the loop termination boundary

The Golden Loop: Predict → Act → Verify

Generator Node — Predict

Queries Gemini, Groq, OpenRouter, or a local Ollama instance depending on the selected provider.

The hard part: Prompting for structured JSON is fragile. If the model returns unstructured prose, the JSON parser breaks the whole pipeline.
Hardening: For Gemini, the official SDK's response_schema with Pydantic forces native JSON output. For Ollama, format="json" is set and a defensive multi-step extractor parses JSON using regex and curly-brace indexing as fallbacks.

Executor Node — Act & Verify

Writes files to the UUID sandbox, runs the compiler, and returns a pass/fail signal with full logs. See Lesson 2 for the full executor detail.

Conditional edge: If tests pass or retry_count ≥ 3, the graph transitions to END. Otherwise it loops back to the generator, injecting the error logs into the repair prompt.

Multi-Language Runtime Challenges

Supporting five runtimes in one engine surfaces environment-specific problems:

  • Windows npx resolution: npx is a CMD script, not a binary. Running it directly raises WinError 2. Fixed by prepending ["cmd.exe", "/c"] dynamically on Windows.
  • TypeScript type-check errors: TypeScript fails compilation in a clean sandbox with no @types/node. Fixed by passing --transpile-only to ts-node, which validates runtime syntax without full type resolution.
  • Antivirus scan overhead: Compiling Go or Java triggers Windows Defender scans that can consume 20+ seconds. Fixed with a 30-second timeout for compiled languages vs. 15 seconds for interpreted ones.

🧠 Retrieval Practice

How does the LangGraph generator node use compiler error logs to self-heal code in subsequent iterations?

Rather than discarding the failed attempt, the healing loop builds on it. Providing the exact failing code alongside its compiler errors lets the LLM act as a targeted editor rather than a blind generator — which produces far higher accuracy repairs.
