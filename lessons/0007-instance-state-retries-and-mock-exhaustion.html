Lesson 7 · Resilience Engineering

Instance-State Retries & Mock-Exhaustion StopIteration

A subtle but production-breaking bug: static decorators that ignore runtime configuration, causing test mocks to exhaust and crash with a cryptic StopIteration. Here is what happens and how to fix it permanently.

The Bug: Static Decorators vs. Instance Variables

When an LLM generates a class with retry logic, it commonly reaches for a static method decorator: 

# ❌ COMMON LLM-GENERATED PATTERN — DO NOT USE
class APIClient:
    DEFAULT_MAX_RETRIES = 3

    @retry(
        stop=stop_after_attempt(DEFAULT_MAX_RETRIES + 1),  # Evaluated at class load
        wait=wait_exponential(multiplier=1, min=1, max=10),
        reraise=True
    )
    def _request(self, method, path, **kwargs):
        ...

Python decorators are class-level constructs evaluated once at module import time. The stop_after_attempt(DEFAULT_MAX_RETRIES + 1) expression is computed when the class loads — not when the method is called. This is the root cause of the bug.

State-Isolation Breach

If the caller instantiates the client with a custom retry count — APIClient(max_retries=2) — the static decorator ignores it entirely. The decorator stays bound to DEFAULT_MAX_RETRIES.

OOP principle broken: Instance configuration passed to __init__ should govern instance behaviour. Static decorators break this by binding configuration at class load time.

The Symptom: Mock Exhaustion & StopIteration

The mismatch surfaces in unit tests as a cryptic StopIteration. Here is why: 

# Test written to expect exactly 3 attempts (max_retries=2)
def test_retries_exceeded():
    client = APIClient(max_retries=2)
    with patch("requests.Session.request") as mock_req:
        mock_req.side_effect = [
            mock_response(429),   # attempt 1
            mock_response(429),   # attempt 2
            mock_response(429),   # attempt 3 — we expect this to raise
        ]
        with pytest.raises(RateLimitError):
            client.list_items()

Why StopIteration Fires

The test provides 3 mock responses, matching the expected 3 attempts. But the static decorator forces DEFAULT_MAX_RETRIES + 1 = 4 attempts. Tenacity triggers a 4th network call.

What happens: unittest.mock calls next() on the depleted side_effect iterator. No elements remain. Python raises StopIteration. Pytest catches it as an unexpected error — not as the RateLimitError the test was asserting.
Diagnosis trap: The error message says StopIteration, not "retry count mismatch". Without knowing this pattern, it reads like a framework bug rather than a configuration isolation problem.

The Fix: Dynamic Retrying

Remove the static decorator. Construct the tenacity controller dynamically inside the method at runtime, reading self.max_retries in the moment the call is made. 

# ✅ CORRECT PRODUCTION PATTERN
def _request(self, method, path, **kwargs):
    url = f"{self.base_url}{path}"

    # Built fresh on every call — reads live instance state
    retrier = Retrying(
        stop=stop_after_attempt(self.max_retries + 1),
        wait=wait_exponential(multiplier=1, min=1, max=10),
        reraise=True,
        retry=retry_if_exception_type((RateLimitError, ServerError)),
        before_sleep=before_sleep_log(logger, logging.INFO)
    )

    return retrier(
        lambda: self._handle_response(
            self.session.request(method, url, timeout=self.timeout, **kwargs)
        )
    )

Why this works

Retrying(...) is called as a regular function inside _request. It runs after __init__ has set self.max_retries. The stop condition reads the live instance value every time — not a class-level constant baked in at import.

Result: Mocks can now be sized exactly to match the configured retry count. StopIteration never fires. Tests are deterministic and trustworthy.

🧠 Retrieval Practice

Why does a static @retry decorator on an instance method cause mock tests to crash with StopIteration when max_retries is overridden at instantiation?

Python evaluates class decorators once at import time. If the test configures a client with fewer retries than the static decorator expects, tenacity fires more calls than the mock has responses for. unittest.mock raises StopIteration on the exhausted iterator — appearing as a mysterious test framework error rather than a configuration mismatch.
