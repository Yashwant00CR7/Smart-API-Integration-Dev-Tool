Lesson 6 · Docker, Runtimes & Cloud Security

Containerised Multi-Runtime Sandboxing & HF Spaces Deployment

Bundling five language runtimes into one Docker image, then deploying it on a cloud platform that runs as a non-root user. Three specific problems — and their solutions.

1. Multi-Runtime Containerisation

To compile Python, JavaScript, TypeScript, Go, and Java tests in a single application, all five runtimes must be present in the execution environment. On a developer's local machine this is manageable. On a cloud host it must be explicitly bundled. 

# Install all language runtimes in one Debian layer
RUN apt-get update && apt-get install -y \
    curl build-essential golang default-jdk \
    && curl -fsSL https://deb.nodesource.com/setup_18.x | bash - \
    && apt-get install -y nodejs \
    && rm -rf /var/lib/apt/lists/*

Why a single-stage image?

All five compilers in one layer means the Python subprocess executor can call go test, javac, node, and pytest directly without any network downloads at runtime.

Result: Test executions are instant, fully offline, and deterministic — no npm downloads, no compiler resolution at generation time.

2. Hugging Face Spaces — Port & User Constraints

Hugging Face Spaces enforces two hard constraints on Docker containers:

  • Port 7860 only. All external traffic is routed exclusively to this port. Uvicorn must bind to 0.0.0.0:7860.
  • UID 1000 non-root execution. Spaces override the container user and run the process as a restricted non-root user with UID 1000.

If the application creates files in directories owned by root — which our UUID sandbox executor does — the runtime raises PermissionError immediately. 

# Create a non-root user matching the UID Spaces enforces
RUN useradd -m -u 1000 user
USER user
ENV HOME=/home/user \
    PATH=/home/user/.local/bin:$PATH
WORKDIR $HOME/app

# Copy all files with ownership assigned to user (not root)
COPY --chown=user . $HOME/app

Why --chown=user matters

Without this flag, files are copied as root. When Spaces runs the container as UID 1000, the process cannot write to those root-owned paths — including the temp/ directory where sandbox test files are created.

Result: All copied files and runtime-created directories are owned by UID 1000, giving the non-root process full write access.

3. Pre-Baking TypeScript — Global npm Install

npx ts-node normally fetches packages dynamically from the npm registry on first use. Inside a cloud container this introduces latency spikes, network timeouts, and npm rate limits on every test run. 

# Pre-install globally as root, before switching to user
RUN npm install -g typescript ts-node

Why install as root before the USER switch?

Global npm installs require write access to /usr/lib/node_modules. This must happen before the USER user instruction drops privileges. After the switch, the globally installed binaries are still accessible as read-execute to UID 1000.

Timing matters: Reversing the order — switching to user first, then running npm install -g — raises a permission error during the Docker build.
Result: TypeScript tests execute offline with zero npm download latency, every time.

🧠 Retrieval Practice

Question 1 — Why will an application that creates files in root-owned directories fail on Hugging Face Spaces?

Hugging Face Spaces strictly enforce UID 1000 for security isolation. If container directories are owned by root, the non-root process cannot read or write to them at runtime — causing application failure even though the build succeeded.

Question 2 — What is the main benefit of running npm install -g typescript ts-node in the Dockerfile before the USER switch?

Pre-baking global dependencies eliminates compilation latency, removes network calls at test time, and ensures the container is fully self-contained and reproducible.
