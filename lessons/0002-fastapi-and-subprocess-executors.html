Lesson 2 · Backend & Subprocess Sandboxing

FastAPI Skeletons & Subprocess Test Runtimes

For an agent to self-heal, it must be able to actually run the code it generates. This lesson covers how the backend is structured and how tests are executed safely in isolated sandboxes.

A self-healing code generator cannot rely on the LLM guessing whether its output is correct. The system needs to execute and test the code in real time. This lesson covers the three pillars that make that possible.

The 3 Pillars of Backend Validation

1. FastAPI Skeletal Core

FastAPI coordinates everything: serves the static dashboard, handles scraping requests, and spins up the LangGraph agent.

Why FastAPI? It is lightweight, uses Pydantic for automatic request schema validation, and handles async loops natively — ideal for long-running generation and scraping requests.
In src/app.py: Static folders are mounted conditionally. The POST /api/analyze route triggers the scraper if a URL is given, or skips the network call entirely if raw text is pasted.

2. Concurrency-Safe UUID Sandboxing

When compiling and running code, concurrent requests cannot share a single temp file. If two users trigger generation simultaneously, writing to the same temp_client.py would cause race conditions and corrupt compiler output.

The danger: Parallel file writes cause race conditions, crashing compiles and mixing test logs between users.
The fix: The executor creates a unique subdirectory per run using a UUID — e.g. temp/run_150754a0-677d-4d46-bf83-49218b1a8555/. Code is written, executed, and the directory is deleted inside that one request's lifecycle.

3. Controlled Subprocess Execution

Tests are executed by spawning an OS subprocess via Python's subprocess.run. This isolates the test runner from the main web server thread.

The danger: An LLM might generate an infinite loop (while True: pass). Without a timeout, that subprocess hangs the server thread indefinitely.
The fix: All processes run with a strict timeout=15 (30 for Go/Java). A TimeoutExpired exception is caught and returned as a failure log to the agent for repair.

The Executor Workflow

1. Code + tests arrive at run_tests()
   │
   ▼
2. Create unique UUID sandbox directory under temp/
   │
   ▼
3. Write client.py and test_client.py inside the sandbox
   │
   ▼
4. subprocess.run(["pytest", "test_client.py"], cwd=sandbox, timeout=15)
   │
   ▼
5. Capture exit code (0 = pass, non-zero = fail) + stdout/stderr logs
   │
   ▼
6. Delete UUID sandbox directory (cleanup)
   │
   ▼
7. Return (success: bool, logs: str) to LangGraph agent

🧠 Retrieval Practice

Why is it critical to enforce a timeout when spawning compiler processes in a self-healing pipeline?

An LLM can occasionally produce logic with recursive infinite loops or long-hanging HTTP calls. Without a timeout, that subprocess blocks a thread forever and eventually crashes the server. The timeout lets us intercept it, capture the hang as an error log, and pass it back to the LLM for repair.
