Lesson 4 · IDE Tool Integration & Protocols

Model Context Protocol & Local IDE Integration

MCP lets IDE agents call this tool directly as a local subprocess. This lesson covers the stdin/stdout transport, the critical stdout corruption problem, and the JSON-RPC handshake lifecycle.

AI assistants like Claude Desktop and Cursor run directly inside developers' editors and can interact with local environments through the Model Context Protocol (MCP) — an open standard by Anthropic. Rather than always using the web dashboard, the tool can be launched as an MCP server and called natively from within the IDE.

Standard I/O Stream Transport

The primary local MCP transport is stdin/stdout. The IDE client launches the server as a subprocess, writes JSON-RPC 2.0 request frames to its stdin, and reads JSON-RPC 2.0 response frames from its stdout.

The Stream Corruption Problem

Because stdout carries the protocol frames, any casual print statement, warning, or library log going to stdout will corrupt the stream. The IDE client will fail to parse it as JSON-RPC and immediately drop the connection.

Real risk: A connection warning from google-genai, or a single print("Generating..."), silently crashes the MCP session.
Solution: On startup, we save a reference to the original sys.stdout and then swap sys.stdout to point to sys.stderr. All prints and library logs go safely to stderr (visible as IDE logs). JSON-RPC responses are written exclusively via the saved original stdout reference.

Protocol Handshake Lifecycle

Step 1 — Handshake

The session opens with a two-message negotiation.

initialize — client sends its protocol version. Server replies with its name, version, and supported protocol version.

initialized — client sends a notification acknowledging the handshake. No server response is expected (notifications do not get responses per JSON-RPC 2.0).

Step 2 — Tool Discovery (tools/list)

The client requests the catalogue of available tools. The server replies with names, descriptions, and full JSON Schema parameter definitions.

Our server exposes two tools:
  • scrape_url — scrapes a documentation page via Firecrawl
  • generate_wrapper — runs the full self-healing LangGraph generation pipeline

Step 3 — Execution (tools/call)

The client calls a tool with a name and arguments object.

Error handling: If an exception occurs, the server sets isError: true inside the result content block rather than letting the exception propagate — keeping the process alive for subsequent requests.

🧠 Retrieval Practice

Why does the MCP server redirect all Python print output to stderr during startup?

Under stdin/stdout transport, stdout is the dedicated JSON-RPC pipe. Any non-JSON text — a print statement, a library warning — corrupts the frame stream and causes the IDE client parser to error. Redirecting stdout to stderr keeps the channel clean while still making logs visible.
