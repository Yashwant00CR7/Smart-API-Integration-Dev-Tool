Lesson 5 · Frontend & Asset Packaging

Serving Static Assets, Client-Side Caching & ZIP Export

Three architecture decisions that keep the tool lightweight: serving the entire frontend from FastAPI, storing history in the browser instead of a database, and packaging downloads entirely in-browser with no server involvement.

For a single-user developer tool or hackathon deployment, complex infrastructure is overkill. This lesson covers three patterns that keep the application stateless, fast, and zero-dependency on external databases or file storage.

1. Programmatic Static Asset Serving

FastAPI's StaticFiles middleware serves the entire React dashboard directly from the public/ directory on the root path. No separate Node server or reverse proxy is needed in production. 

public_path = os.path.abspath(
    os.path.join(os.path.dirname(__file__), "..", "public")
)

if os.path.isdir(public_path):
    # html=True auto-serves index.html at "/" and subdirectory paths
    app.mount("/", StaticFiles(directory=public_path, html=True), name="public")
else:
    @app.get("/")
    def read_root():
        return {"status": "Backend running. Static assets not yet built."}

Why the conditional check?

In development, the public/ directory may not exist yet (frontend not built). Without the guard, FastAPI raises an error at startup instead of gracefully falling back to a JSON health route.

Result: A single python main.py command serves both the API and the full UI — one process, one port, one Dockerfile.

2. Stateless Backend via Browser localStorage

Rather than storing integration history or API keys in SQLite or PostgreSQL, all persistent state lives in the developer's own browser. The backend is completely stateless. 

// Save an integration record
const record = { id: "integration_1720000000000", title: "stripe.com", code: "..." };
localStorage.setItem("api_integrations_history", JSON.stringify([record]));

// Retrieve on page load
const history = JSON.parse(localStorage.getItem("api_integrations_history") || "[]");

// API keys — session-scoped (clears when tab closes)
sessionStorage.setItem("credentials_gemini_key", "AIzaSy...");

Why not a database?

Security: API keys never leave the developer's machine. No server-side credential storage means no server-side credential breach surface.
Architecture benefit: The backend has no state to synchronise, backup, or migrate. Resetting the tool is as simple as clearing browser storage.

3. Client-Side ZIP Export with JSZip

Normally, packaging files into a .zip requires the server to compress them and stream the binary back. JSZip does the entire operation in the browser, eliminating server-side CPU, memory, and storage overhead entirely. 

import JSZip from "jszip";

const zip = new JSZip();
zip.file("client.py",       generatedClientCode);
zip.file("test_client.py",  generatedTestsCode);
zip.file("README.md",       generatedReadme);

const blob = await zip.generateAsync({ type: "blob" });

const a = document.createElement("a");
a.href = URL.createObjectURL(blob);
a.download = "api-integration-python.zip";
a.click();
URL.revokeObjectURL(a.href);

Client-Side Packaging

JSZip assembles the zip byte representation in a Web Worker, then creates an object URL for instant download.

Result: Instant downloads, no server storage cost, no server CPU overhead, and perfect alignment with the stateless backend architecture.

🧠 Retrieval Practice

Question 1 — Why is browser localStorage preferred over a server database for storing developer API keys in this tool?

Keeping keys in the browser means no server-side credential storage — and therefore no server-side credential breach surface. The backend stays stateless and the developer retains full control of their keys.

Question 2 — What does setting html=True on FastAPI's StaticFiles mount accomplish?

html=True makes StaticFiles search for and serve index.html as the directory index default — which is exactly how single-page applications expect to be served.
